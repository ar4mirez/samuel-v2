# RFD 0011: OCI plugin tier + network policy (v2.3)

## Summary

v2.0 scaffolded the OCI tier but left three things unfinished:
container runtime detection, agent containerization, and the
unresolved open question on network policy granularity. v2.3 lands
all three.

- `internal/plugin/oci/runtime.go` resolves the engine via rootless
  Podman → root Podman → Docker, with `SAMUEL_RUNTIME` as an
  explicit override.
- The manifest's `[oci]` block gains `entrypoint`, `workdir`,
  `cpu_quota`, and `memory_limit`; images MUST be digest-pinned at
  parse time.
- A new `internal/plugin/oci/policy` package implements
  deny-by-default outbound network policy with per-call consent
  prompts, persistence at `~/.samuel/policy/network.toml`, and an
  append-only audit log.
- A userspace proxy at `internal/plugin/oci/policy/proxy` brokers
  every outbound request; the launcher pins `--network=none` so the
  proxy is the only egress.
- `samuel policy list / reset / prompt / preauth` give operators
  control of the consent store, including a CI-friendly preauth path.
- `samuel run --sandbox=oci` runs the agent inside a container with
  samuel-managed env (host mode preserved for development).
- A reference plugin, `samuel-claude-code-oci`, packages Claude Code
  as an OCI plugin and validates the end-to-end flow.

## Motivation

The wiki's open question on OCI network policy granularity has
blocked the tier since v2.0. With agent containerization pending (the
rc.16 HOME/PATH-passthrough bug surfaced the cost of extending the
host-exec environment), we now have a real use case. Three options
were on the table:

1. **Host allowlist only.**
2. **Regex allowlist.**
3. **Deny-by-default with per-call consent.**

This RFD records the decision and the implementation.

## Decision: deny-by-default with explicit allowlist + per-call consent prompts

Rationale:

- **Host-only** is too permissive. A manifest declaring `github.com`
  can exfiltrate via DNS lookups, redirects, or sibling hosts. The
  trust model becomes "the manifest tells the truth about its
  destinations" — a brittle invariant.
- **Regex** is too clever. Patterns are hard to audit and encourage
  overly broad matches. The cost of getting it wrong is silent
  exfiltration.
- **Deny-by-default + consent** matches the rest of Samuel's UX —
  signed-by-default, capability-declared. The first time a plugin
  reaches a non-allowlisted host, the user picks one of four
  outcomes: allow once, always allow, deny, deny forever. Decisions
  persist; the user is never re-prompted for the same `(plugin,
  host)` pair.

The proxy enforcement is the load-bearing piece: with
`--network=none` at the container boundary and `HTTP_PROXY=unix:///samuel-proxy`
inside, the proxy is the only egress. DNS lookups flow through the
proxy too; raw-IP destinations never match a hostname allowlist and
are blocked automatically.

## Options considered

### Host allowlist only

**Pros:** simplest implementation; no UX cost (no prompts);
deterministic.

**Cons:** trust model is fragile. Once `github.com` is in the
allowlist, the plugin can do anything reachable at that host
(GitHub's REST API, GitHub raw content, repos under attacker control,
…). The user has no signal when behavior drifts.

### Regex allowlist

**Pros:** more granular than host-only; supports `*.example.com`
patterns cleanly.

**Cons:** regex is unreadable as an audit trail. Plugin authors
write overly broad patterns to avoid edge cases (`^.*\.example\.com.*$`
is a real footgun). Users reading the manifest can't quickly answer
"is this safe?" — the cognitive cost of a regex review is far
higher than a list of literal hosts. We already get the wildcard
escape hatch via `*.example.com` matching in the chosen design,
without exposing arbitrary regex.

### Deny-by-default with consent (chosen)

**Pros:** explicit; auditable; resilient. Bad behavior surfaces
immediately as a prompt; users can deny without restarting. CI flows
get clean primitives (`SAMUEL_POLICY=deny-all` + `samuel policy preauth`).

**Cons:** UX friction on first run. Mitigated by the manifest
allowlist (the common case — declared hosts auto-allow) and the
persistent consent store (one decision lasts forever).

## Trade-offs accepted

- **Proxy overhead.** Every request crosses the unix socket; for
  hot loops this can be measurable. Document the perf characteristics
  and offer `--policy-mode=manifest-only` (post-v2.3, security
  review required) for high-throughput plugins.
- **Userspace DNS.** The proxy does its own host resolution. We
  could let the container's libc resolve and only enforce at the IP
  level, but that loses the raw-IP-block invariant. Chosen because
  the security gain outweighs the proxy code complexity.
- **Manifest signing is image signing.** OCI signing targets the
  image digest, not a blob. `samuel install` validates the digest
  in `[oci].image` against the cosign bundle subject. Documented in
  `docs/plugin-authors/signing.md`.

## Implementation pointers

- `internal/plugin/oci/runtime.go` — detection + image pull.
- `internal/plugin/oci/launcher.go` — `BuildRunArgs`, capability
  mount derivation, resource limits.
- `internal/plugin/oci/policy/policy.go` — engine, store, modes.
- `internal/plugin/oci/policy/proxy/proxy.go` — userspace HTTP/HTTPS
  proxy.
- `internal/commands/policy.go` — `samuel policy` subcommands.
- `internal/commands/update_agents.go` — `samuel update --agents`
  agent image pinning.
- [`samuelpkg/samuel-claude-code-oci`](https://github.com/samuelpkg/samuel-claude-code-oci) — reference plugin (split out 2026-05-15).

## Outcome

Filled post-implementation.

- Manifest schema bumped to `plugin.v2.3.json` with digest-pinning
  enforcement.
- `samuel doctor` reports `oci-runtime` health (engine + version +
  cache size).
- v2.3.0 ships OCI sandbox as opt-in (`samuel run --sandbox=oci`);
  v2.4 flips the default after the 2-week soak window.
